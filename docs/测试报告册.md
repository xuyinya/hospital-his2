文档版本：v1.0

一、测试概述

项目背景：东软云医院系统是基于 B/S 架构的医院门诊信息管理系统，实现挂号、医生、患者、病历、处方、药品、检查、检验、处置、收费等门诊业务全流程信息化管理。技术栈为 Spring Boot 3.2.5 + Vue 3 + MySQL 8.0。

测试目的：验证后端核心 Service 层的业务逻辑正确性；验证 JWT 令牌生成、解析、验证的安全机制；验证 CRUD 操作的数据完整性和参数透传；验证分页查询的边界条件和多条件过滤；验证挂号状态流转的正确性。

测试范围：本次测试覆盖 4 个模块，共计 30 个测试用例。JWT 安全模块 5 个用例（纯单元测试，无依赖），患者管理模块 8 个用例（Mock Mapper 模拟 Mapper 层，不连接真实数据库），药品管理模块 7 个用例（Mock Mapper 模拟 Mapper 层，不连接真实数据库），挂号管理模块 10 个用例（Mock Mapper 模拟 Mapper 层，不连接真实数据库）。

测试策略：JwtUtil 采用纯单元测试方式，不依赖 Spring 容器和数据库，手动构造配置对象测试 JWT 全生命周期。Service 层使用 Mockito 框架模拟 Mapper 接口，不依赖真实数据库，验证业务逻辑对 Mapper 的调用是否正确以及返回值处理是否正确。

五、缺陷记录

本次测试共发现 4 个缺陷，覆盖严重、一般、轻微三个级别，均已修复并验证通过。

缺陷 BUG-001（严重）：药品管理模块的修改接口存在字段缺失问题。DrugMapper.update 方法的 SQL 语句缺少 status=#{status} 字段，导致管理员通过编辑弹窗将药品状态改为"停用"并保存后，数据库中的 status 字段并未被更新，停用操作表面成功但实际无效。修复方式为在 UPDATE 语句中补上 status=#{status}，文件位于 backend/src/main/java/com/neusoft/hospital/mapper/DrugMapper.java 第 33 行。

缺陷 BUG-002（严重）：患者自助挂号时，挂号费用的计算逻辑放在前端实现，客户端可以篡改请求中的 regFee 参数，存在安全隐患。测试发现通过浏览器开发者工具修改请求体的 regFee 值后，后端未做二次校验直接存入数据库，可能导致挂号费用被恶意修改。修复方式为在 RegistrationController.selfRegistration 方法中，以后端根据 regType 字段重新计算费用为准，忽略客户端传入的 regFee 值，普通号固定 25 元，专家号固定 50 元，文件位于 backend/src/main/java/com/neusoft/hospital/controller/RegistrationController.java 第 56 至 63 行。

缺陷 BUG-003（一般）：RegistrationServiceImpl 和 PrescriptionServiceImpl 缺少 @Transactional 事务注解。在并发场景下，如果新增挂号或保存处方过程中发生异常，已执行的数据库操作不会自动回滚，可能导致数据不一致。例如新增挂号时插入了 registration 记录，但后续关联操作失败，数据库中会留下脏数据。修复方式为在两个 Service 实现类上添加 @Transactional 注解，确保业务操作的事务原子性，文件位于 backend/src/main/java/com/neusoft/hospital/service/impl/RegistrationServiceImpl.java 第 18 行和 PrescriptionServiceImpl.java 第 20 行。

缺陷 BUG-004（轻微）：RegistrationMapper 中 insert 方法的 SQL 语句只插入 patient_id、doctor_id、dept_id、reg_type、reg_fee 五个字段，不支持 chief_complaint（主诉）和 present_illness（现病史）字段的写入。患者自助挂号时填写的症状描述信息无法保存到数据库，医生接诊时看不到患者自述内容。修复方式为在 INSERT 语句中补充 chief_complaint 和 present_illness 两个字段，文件位于 backend/src/main/java/com/neusoft/hospital/mapper/RegistrationMapper.java 第 21 至 22 行。


六、测试结论

本次测试针对 HIS 系统的核心业务模块进行了 30 个单元测试用例的验证，全部通过，通过率 100%。测试覆盖了 JWT 安全认证、患者管理、药品管理、挂号管理等关键业务领域，验证了 CRUD 操作的正确性、状态流转的可靠性以及分页查询的边界处理。测试过程中发现 4 个缺陷，其中严重缺陷 2 个、一般缺陷 1 个、轻微缺陷 1 个，已全部修复并验证通过。

后续建议：一是扩展测试覆盖范围，当前测试聚焦于 Service 层，建议增加 Controller 层的接口测试（如 MockMVC）和集成测试（连接测试数据库）。二是建议将测试纳入 CI/CD 流程，每次提交自动运行 mvn test。三是建议创建通用的测试数据工厂类，减少各测试类中重复的实体构造代码。


七、系统运维与维护

7.1 开发环境搭建

后端运行依赖 JDK 21、Maven 3.9.4 和 MySQL 8.0。首次运行前需执行 hospital_his.sql 脚本初始化数据库和测试数据。Maven 本地仓库配置在 D 盘路径 D:\java\JavaWebMaven\apache-maven-3.9.4\mvn_repo，已在 Maven 全局 settings.xml 中配置。启动后端时需指定 JDK 路径，命令为 JAVA_HOME="C:\Program Files\Java\jdk-21" mvn clean package -DskipTests 编译打包，然后 java -jar target/hospital-his-1.0.0.jar 启动服务，端口 8080。前端在 frontProject 目录下执行 npm install 安装依赖后，运行 npm run dev 启动开发服务器，端口 3000，通过代理访问后端接口。

7.2 构建配置说明

pom.xml 中关键配置项包括 surefire 插件版本 3.5.3 和 junit-platform 版本 1.11.4，这两个版本需匹配使用，否则测试执行会报错。数据库连接配置在 application.yml 中，当前用户名 root，密码 12345，连接地址 localhost:3306/hospital_his。JWT 密钥为 NeusoftHospitalHIS2026SecretKeyForJWTTokenGeneration，过期时间为 86400000 毫秒（24 小时）。MyBatis 的 mapper 映射文件位于 resources/mapper 目录下，共 11 个 XML 文件，与 Mapper 接口配合使用。

7.3 数据库维护

数据库名称 hospital_his，共有 12 张核心业务表和 1 张系统用户表。初始化脚本包含 8 个科室、8 名医生、8 种药品、5 位患者、7 条挂号记录、3 份病历、1 条处方及明细、3 条检查、4 条检验、2 条处置、7 条收费记录以及 9 个系统用户账户。当需要重置测试数据时，重新执行 hospital_his.sql 即可。数据库表之间通过外键约束保证数据完整性，删除医生或患者时需先处理关联的挂号记录。

7.4 测试执行说明

测试代码位于 src/test/java 目录下，当前有 4 个测试类共 30 个测试用例。执行全部测试使用 mvn test 命令，执行单个测试类使用 mvn test -Dtest=类名。测试采用 Mockito 框架模拟 Mapper 层，不需要数据库连接即可运行。测试报告输出在 target/surefire-reports 目录下。后续增加测试类时，类名需以 Test 结尾，放在对应包的 test 目录下。

7.5 日常维护注意事项

系统用户账户通过 SQL 脚本初始化，密码使用 BCrypt 加密存储，默认密码为 123456，管理员账户 admin 的密码为 admin123。若需新增预置账户，需要先通过 BCrypt 加密密码再写入 SQL。患者密码由 DataInitializer 在系统启动时自动初始化，已有密码不会被覆盖。需要注意的是，surefire 插件和 junit-platform 的版本必须匹配，否则测试执行会因找不到类而失败。


八、系统细节问题修改

8.1 首页统计数据改为全量统计

原 DashboardController 的统计接口按当日（Asia/Shanghai 时区）过滤数据，导致演示时看不到历史数据。修改后将统计范围改为全部数据，不再按日期过滤，方便在演示环境中展示完整的运营数据。首页统计卡片的标签也相应从"今日挂号""今日就诊""今日收入"改为"挂号总数""已就诊数""总收入"。涉及文件为 backend/src/main/java/com/neusoft/hospital/controller/DashboardController.java 和 frontProject/src/views/dashboard/index.vue。

8.2 系统用户数据从代码迁移到 SQL

原 DataInitializer 中通过 Java 代码创建系统管理员和医生账户，每次启动时检查是否存在再创建。重构后将 9 个系统用户账户的 INSERT 语句直接写在 hospital_his.sql 中，启动时无需重复检查。DataInitializer 简化为仅负责为已有患者设置默认登录密码，不再管理系统用户。涉及文件为 backend/src/main/java/com/neusoft/hospital/config/DataInitializer.java 和 backend/hospital_his.sql。

8.3 VO 层合并到 Entity 重构

原项目为每个业务模块分别定义了 Entity（数据库映射）和 VO（视图对象），两者之间通过继承或手动转换传递数据，导致代码冗余。重构后删除 entity/vo 目录下的 9 个 VO 类，将 VO 中的关联字段（如 patientName、doctorName、deptName 等）直接合并到 Entity 中。Mapper 方法名去掉 VO 后缀，Service 和 Controller 的返回类型统一改为 Entity。本次重构共涉及 64 个文件，新增 244 行，删除 546 行。

8.4 挂号记录增加主诉和现病史字段

原 registration 表只记录患者 ID、医生 ID、科室 ID、挂号类型和费用等基本信息，不支持患者描述症状。为配合患者自助挂号功能，在 registration 表中新增 chief_complaint（主诉，最长 500 字）和 present_illness（现病史，最长 1000 字）两个字段。后端 Registration 实体同步增加对应属性，Mapper 的 insert 和 select 语句相应更新，前端医生接诊工作台可查看患者自述内容。涉及文件为 Registration.java、RegistrationMapper.java、RegistrationMapper.xml、hospital_his.sql。

8.5 PrescriptionController 返回值改为处方 ID

原新增处方接口返回空结果，前端无法获取新创建的处方 ID，导致无法继续添加处方明细。修改后将 PrescriptionController.add 方法的返回值从 Result<Void> 改为 Result<Long>，返回自动生成的处方 ID。前端医生接诊工作台在创建处方后获得处方 ID，再逐条添加药品明细，完成开药流程。涉及文件为 backend/src/main/java/com/neusoft/hospital/controller/PrescriptionController.java。

8.6 Maven 构建配置兼容性修复

原 pom.xml 中 maven-surefire-plugin 使用 Spring Boot 父 POM 管理的默认版本 3.1.2，但本地 Maven 仓库缺少该版本的 surefire-junit-platform 依赖，导致 mvn test 执行时报 NoSuchFileException 错误。修复方式为覆盖 surefire 版本为 3.5.3（本地仓库已有的版本），同时显式声明 junit-platform-launcher 和 junit-platform-engine 为 1.11.4 版本，与 surefire 3.5.3 兼容。涉及文件为 backend/pom.xml。
